# ゲームシステム概要

この文書は、Dungeon Match Heroes の現在のゲームシステムを大まかに整理したものです。実装の細部ではなく、ゲームがどのような要素で構成され、各システムがどのように連携しているかを把握するための資料です。

## 1. 基本コンセプト

Dungeon Match Heroes は、マッチパズルと RPG 戦闘を組み合わせたゲームです。プレイヤーは 7x7 のブロック盤面を操作し、発生したマッチによって味方パーティの攻撃、防御、回復、成長を行います。

通常のマッチ 3 のようにブロックを入れ替えるのではなく、最下段のブロックを直接破壊することが特徴です。破壊後にブロックが落下し、補充された結果としてマッチが発生します。

## 2. ゲーム全体の流れ

ゲームは以下の流れで進行します。

1. タイトル画面からゲーム開始。
2. Main シーンで敵ウェーブが出現。
3. プレイヤーが最下段のブロックを破壊。
4. ブロックが落下、補充され、マッチ判定が行われる。
5. マッチ結果が戦闘アクションに変換される。
6. 味方と敵のアクションがキューで順番に処理される。
7. 敵を全滅させるとウェーブクリア。
8. 宝箱イベントや Tips 表示を挟んで次のウェーブへ進む。
9. プレイヤー HP が 0 になると GameOver 画面へ遷移する。

## 3. パズルシステム

盤面は 7x7 のグリッドです。プレイヤーが操作できるのは最下段のブロックのみです。

ブロックをクリックすると、そのブロックが破壊されます。その後、上のブロックが下へ落ち、空いた場所に新しいブロックが補充されます。補充後に同じ種類のブロックが 3 個以上つながっているとマッチとして消滅します。

マッチは横方向と縦方向の 3 連以上から検出されます。実際に消える対象は、検出された 3 連だけでなく、同じ種類で隣接しているクラスター全体です。

マッチが消滅すると再び落下と補充が発生し、追加のマッチがあれば連鎖として処理されます。

## 4. ブロック種別

現在のブロック種別は以下です。3つ揃えることで効果を発揮する通常のブロック群を、総称して **Item Blocks（アイテムブロック）** と呼びます。

| 種別 | 主な効果 |
| --- | --- |
| Sword | 前方の敵に物理攻撃を行う |
| Shield | シールドを加算する |
| Magic | 敵全体に魔法攻撃を行う |
| Heal | HP を回復する。HP 満タン時は EXP に変換される |
| Gem | EXP を獲得する |
| Key | 鍵を獲得する。既に鍵を持っている場合は EXP に変換される |
| ? / Junk | 通常マッチでは消えない特殊ブロック。ユーザー向けには「? ブロック」と呼び、開発上は Junk と呼ぶ |

各ブロックは、共通のベーススプライト、種別ごとのアイコン、種別ごとの tint によって表示されます。

## 5. ? ブロック / Junk

この特殊ブロックは、ユーザー向けには「? ブロック」と呼びます。開発上の内部名としては Junk を使います。

? ブロックは盤面を詰まらせる特殊ブロックです。通常ブロックのようにマッチして消えることはありません。

? ブロックの基本ルールは以下です。

- 通常ブロックを手動で破壊した後の補充では、? ブロックが出現する可能性がある。
- ? ブロックを手動で破壊すると、次の補充では通常ブロックが出る。
- マッチした通常ブロック群に隣接している ? ブロックは誘爆して一緒に消える。
- 誘爆した ? ブロックは、元になったマッチの効果を少し強化する。

この仕組みにより、プレイヤーは単にマッチを作るだけでなく、? ブロックを処理して盤面を整える判断も求められます。

## 6. 戦闘システム

パズルで発生したマッチは、CombatManager によって戦闘アクションへ変換されます。マッチ数が多いほど効果が大きくなり、隣接する ? ブロックの誘爆数も効果量に反映されます。

主な戦闘効果は以下です。

- Sword: Fighter が前方の敵を攻撃する。
- Magic: Mage が敵全体を攻撃する。
- Heal: パーティ HP を回復する。
- Shield: Tank を中心にシールドを加算する。
- Gem: EXP を獲得する。
- Key: 鍵を獲得する、または EXP を獲得する。

プレイヤーアクションと敵アクションはキューで管理されます。これにより、アニメーション、効果音、ダメージ処理が重ならず、順番に解決されます。

## 7. 主人公パーティ

主人公パーティは Fighter、Mage、Tank の 3 人で構成されています。プレイヤーが直接キャラクターを選択して行動させるのではなく、パズルで発生したマッチ種別に応じて各キャラクターの役割が発動します。

| キャラクター | 主な役割 | 対応する主なブロック |
| --- | --- | --- |
| Fighter | 前方の敵への単体物理攻撃 | Sword |
| Mage | 敵全体への魔法攻撃 | Magic |
| Tank | シールド獲得と防御演出 | Shield |

パーティ全体で HP、Shield、EXP、Key を共有します。HP が 0 になるとゲームオーバーになります。Shield は物理攻撃を受けたときに消費され、魔法攻撃は防げません。

Heal、Gem、Key は特定の一人だけではなく、パーティ全体の状態に作用します。Heal は HP 回復、Gem は EXP 獲得、Key は宝箱を開けるための鍵として扱われます。

各キャラクターには Animator と CharacterVisuals が関連付けられており、攻撃、被ダメージ、回復、シールド、ゲームオーバー時のシルエット化などの演出に使われます。

## 8. 敵システム

敵はウェーブごとに出現します。現在の敵定義には、Slime、Skeleton、ZombieMage、Orc、Golem などがあります。

敵はそれぞれ HP、攻撃力、攻撃間隔、魔法攻撃かどうかといったパラメータを持ちます。これらは GameBalanceData に定義されています。

敵は時間経過で攻撃アクションをキューに追加します。物理攻撃は Shield によって軽減または無効化できますが、魔法攻撃は Shield では防げません。

敵の並び順も意味を持ちます。前方の敵ほど攻撃しやすく、後方の敵は攻撃間隔に補正がかかります。

## 9. ウェーブと進行

ウェーブ開始時、CombatManager は現在のウェーブ数に応じた予算を計算し、その予算内で敵を選んで出現させます。

ウェーブが進むほど予算が増え、強い敵が出現しやすくなります。敵選択には重み付けがあり、序盤は弱い敵が中心、後半は強い敵が混ざりやすくなる設計です。

敵を全滅させるとウェーブクリアになります。ウェーブクリア後は、一定確率で宝箱イベントが発生し、その後 Tips を表示して次のウェーブへ進みます。

## 10. 成長システム

プレイヤーは EXP を獲得してレベルアップします。EXP は主に Gem、余剰 Heal、重複 Key、敵撃破、宝箱報酬などから得られます。

レベルアップすると最大 HP と攻撃力が上昇し、HP は全回復します。レベルアップ時には専用の表示と効果音が再生されます。

必要 EXP、HP 成長、攻撃力成長などの値は GameBalanceData によって調整できます。

## 11. 宝箱と鍵

ウェーブクリア後、一定確率で宝箱イベントが発生します。

プレイヤーが Key を持っている場合、宝箱を開けてボーナス EXP を獲得できます。Key は消費されます。Key を持っていない場合、宝箱は開けられず、報酬は得られません。

Key は 1 個までしか保持できません。既に Key を持っている状態で Key マッチを発生させると、追加の Key ではなく EXP に変換されます。

## 12. UI システム

UI は UI Toolkit で構成されています。主な画面と要素は以下です。

- Title: タイトル画面とゲーム開始。
- Main: HP、Shield、EXP、Key、通知、Tips、宝箱オーバーレイ。
- GameOver: 最終レベル、EXP、突破ウェーブ数の表示。
- Editor UI: バランスシミュレーション用のエディタ画面。

Main HUD は CombatManager が直接 VisualElement を参照して更新します。パズルや戦闘の結果は、通知ラベル、戦闘数値、中央メッセージとして表示されます。

## 13. 音と演出

AudioManager はゲーム内の効果音を管理します。SEType ごとに AudioClip を登録し、必要に応じてランダムピッチや AudioMixerGroup の振り分けを行います。

主な演出には以下があります。

- ブロック破壊、マッチ消滅、落下着地の効果音とエフェクト。
- ショックウェーブやパーティクル。
- カメラシェイク。
- 味方と敵の攻撃アニメーション。
- ダメージ時のフラッシュと揺れ。
- 戦闘数値ポップアップ。
- レベルアップ、勝利、敵接近、宝箱、ゲームオーバーの演出。

## 14. データと調整項目

主な調整データは GameBalanceData に集約されています。

調整できる代表的な項目は以下です。

- 基準マッチ数。
- ? ブロックの効果換算値。
- プレイヤーの HP、攻撃力、成長率。
- 必要 EXP と増加量。
- Gem、宝箱、Shield、Heal などの効果量。
- 敵の HP、攻撃力、攻撃間隔、魔法属性。
- ウェーブ予算と敵選択の重み付け。

GridManager 側では、ブロックの出現重み、手動補充時の ? ブロック出現率、ブロック色、アイコンサイズなどを調整できます。

## 15. シーン構成

現在の主なシーンは以下です。

| シーン | 役割 |
| --- | --- |
| Title | ゲーム開始画面 |
| Main | パズルと戦闘を行うメインゲーム画面 |
| PersistentSystems | AudioManager などの共通システム |
| GameOver | 最終結果表示とタイトルへの戻り |

PersistentSystemsLoader により、必要な共通システムがシーンをまたいで利用できるようになっています。

## 16. 実装上の中心コンポーネント

| コンポーネント | 役割 |
| --- | --- |
| GridManager | グリッド、入力、落下、補充、マッチ判定 |
| CombatManager | 戦闘、ウェーブ、UI 更新、宝箱、進行管理 |
| EnemyUnit | 敵の HP、攻撃タイマー、被ダメージ処理 |
| GameBalanceData | バランス値と敵定義 |
| AudioManager | 効果音管理 |
| CharacterVisuals | フラッシュ、揺れ、色変化などのキャラクター演出 |
| RowHighlighter | 操作可能な最下段の強調表示 |
| StartHintController | 初回操作ヒントの表示制御 |
| TitleScreenController | タイトル画面制御 |
| GameOverScreenController | ゲームオーバー画面制御 |
| PersistentSystemsLoader | 共通システムシーンの読み込み |

## 17. 現在の設計方針

全体としては、Manager 中心の構成です。GridManager がパズルロジックを担当し、CombatManager が戦闘と進行を統括します。UI は UI Toolkit、描画は URP、入力は New Input System の直接参照を使っています。
