# 開発履歴

この文書は、Dungeon Match Heroes がここまでどのように開発されてきたかを大まかにまとめたものです。過去の詳細な作業プランやアシスタントログを削除しても、プロジェクトの流れを把握できるようにすることを目的としています。

## 1. 初期プロトタイプ

最初は、Unity 上で動作する小規模なマッチパズルの試作として始まりました。盤面は 7x7 のグリッドで、一般的なマッチ 3 のようにブロックを入れ替えるのではなく、最下段のブロックを直接クリックして破壊する操作にしました。

ブロックを破壊すると上のブロックが落下し、空いた場所に新しいブロックが補充されます。その結果として 3 個以上の同種ブロックが並ぶとマッチとして消滅する、という基本ループをまず実装しました。

この段階で、Sword、Shield、Magic、Heal、Gem、Key と、通常マッチでは消えない特殊ブロックが定義されました。この特殊ブロックは現在、ユーザー向けには「? ブロック」、開発上は Junk と呼ぶ方針になっています。初期の目的は、クリック、落下、補充、マッチ検出、連鎖が気持ちよく動くかを確認することでした。

## 2. ? ブロック / Junk のルール追加

次に、通常のマッチでは消せない特殊ブロックとして、現在の「? ブロック」が追加されました。これは盤面を詰まらせる要素であり、現在のパズル性を作る中心的な仕組みになっています。開発上の内部名は Junk です。

現在の ? ブロックのルールはおおよそ次のようになっています。

- 通常ブロックを手動で破壊すると、補充時に ? ブロックが出現することがある。
- ? ブロックを手動で破壊すると浄化扱いになり、次の補充では通常ブロックが出現する。
- マッチしたブロック群に隣接する ? ブロックは誘爆して一緒に消える。
- 誘爆した ? ブロックは、元になったマッチの効果を少し強化する。
- 手動補充時の ? ブロック出現率は GridManager の Inspector から調整でき、デフォルトは 100%。

この変更により、単にマッチを作るだけでなく、盤面に溜まった ? ブロックをどう処理するかが重要になりました。

## 3. ブロックビジュアルの改善

初期のブロック表示は、その後生成したピクセルアート風のアセットに置き換えられました。現在は、石板のような共通ベーススプライトの上に、ブロック種別ごとのアイコンを重ねる構成になっています。

視認性を高めるために、以下の調整も行われました。

- Gem を水色に変更。
- Key を黄色に変更。
- ベース上のアイコンサイズを拡大。
- ブロックのベース自体にも種別ごとの tint を付与。
- Block_Base を少し大きく、正方形に近い形へ調整。

これらは、プレイ中にブロック種別を素早く判別しやすくするための変更です。

## 4. 戦闘プロトタイプ

パズル部分が動くようになった後、RPG 風の戦闘システムが接続されました。マッチが発生すると、その内容が CombatManager に渡され、プレイヤー側の戦闘アクションとして処理されます。

各ブロックは次のような効果に対応しています。

- Sword は前方の敵に攻撃する。
- Magic は敵全体に魔法攻撃を行う。
- Heal は HP を回復し、HP が満タンの場合は EXP に変換される。
- Shield は物理攻撃を防ぐシールドを加算する。
- Gem は EXP を獲得する。
- Key は鍵を獲得し、既に鍵を持っている場合は EXP に変換される。

戦闘アクションはキューに積まれ、アニメーションや演出が重ならないよう順番に処理されます。一方で、プレイヤーのマッチ結果は即時性を重視してキューの先頭に追加されるようになっています。

## 5. 敵ウェーブと成長要素

戦闘は単発の敵とのやり取りから、複数ウェーブを進める構成へ拡張されました。敵の定義やバランス値は GameBalanceData という ScriptableObject に分離され、コードを直接変更せずに調整できる形になっています。

現在の進行要素には、以下のものがあります。

- プレイヤーレベル、最大 HP、攻撃力、必要 EXP。
- Slime、Skeleton、ZombieMage、Orc、Golem などの敵定義。
- ウェーブごとの予算に基づく敵出現システム。
- ウェーブが進むほど強い敵が選ばれやすくなる重み付け。
- 前衛/後衛の位置による敵の攻撃間隔補正。
- GameOver 画面へ渡す最終レベル、EXP、突破ウェーブ数の記録。

また、バランス調整を支援するためのエディタ用シミュレーションツールも追加されています。

## 6. UI と画面遷移

UI には UI Toolkit を使用する方針になり、Title、Main、PersistentSystems、GameOver というシーン構成が整えられました。

主な UI 実装は以下です。

- タイトル画面と開始操作。
- Main 画面の HUD。
- HP、Shield、EXP、Key 状態の表示。
- マッチ時の通知と戦闘数値ポップアップ。
- 最初の操作を促すヒント表示。
- ウェーブ開始、勝利、レベルアップなどの中央メッセージ。
- ウェーブ間の宝箱イベントと Tips 表示。
- 最終結果を表示する GameOver 画面。

SceneTransitionController や PersistentSystemsLoader により、シーンをまたいで共通システムを扱える構成になっています。

## 7. 音と演出の強化

AudioManager が追加され、ゲーム全体で使う効果音を一元管理するようになりました。AudioSource のプール、SEType による効果音の参照、AudioMixerGroup への振り分け、ランダムピッチ再生などを担当しています。

視覚演出も段階的に追加されました。

- ブロック破壊、ひび割れ、衝撃波、落下、マッチ消滅の演出。
- カメラシェイク。
- キャラクターのフラッシュ、揺れ、シルエット化。
- 味方と敵の攻撃アニメーション。
- 戦闘ダメージ数値とアクション通知。
- レベルアップ、勝利、敵接近、宝箱、ゲームオーバーの演出。

これらの追加により、単なるパズル試作から、ゲームとして一通り遊べる縦切りの状態になりました。
